1. Initiative Overview

The Redemption Store initiative gives partner sellers a way to convert the reward points and credits they earn through incentives, training, activities, and journeys into real-world value — cash payouts via XTRM, or non-cash rewards (gift cards, vouchers) via Xoxoday.

Before this initiative, TenXEngage tracked reward balances as a simple number with no immutable audit trail, no vendor routing, and no self-service configuration for Client Admins. The Redemption Store replaces that with a full financial-grade system: dual wallets, an immutable ledger, a configurable catalog, and a complete end-to-end redemption flow.

Vendor transparency rule: Partners never see vendor names (XTRM, Xoxoday) in any UI or API response. Routing happens transparently behind the scenes based on the catalog item's category.

3. F-01 — Wallet & Ledger Foundation Done

What this delivered: Every partner now has a real financial wallet. Every balance movement is recorded as an immutable ledger entry. The nav bar shows the partner's live available balance on every page.

3.1 Dual Wallet Model

The old reward_balances table was a simple accumulator — a single number per user per currency. It was renamed to reward_wallets and extended with two concepts:

Individual Wallet

One per PARTNER_SELLER per currency. Earns from personal incentives, training, activities, journeys, deals.

Company Wallet

One per partner_company per currency. Pooled across all sellers in the company. Accessible by PARTNER_ADMIN only.

Each wallet tracks two balance fields:

availableBalance

The amount the partner can actually spend today. Only this balance is shown in the nav widget and the catalog.

reservedBalance

Amount locked against a pending redemption (submitted but not yet fulfilled). Partner cannot spend this.

3.2 Balance Lifecycle

Balances never change directly. Every change goes through WalletService, which writes a LedgerEntry record first, then updates the wallet totals — all in one database transaction. This makes the ledger the authoritative audit trail.

3.3 Earning Integration

The existing RewardGrantService — which handles all earning sources — was updated to route every credit through WalletService.credit(). It does not bypass the ledger.

Idempotency: If the same earning event is delivered twice (e.g., a retry), the second credit is silently skipped. The unique index on (wallet_id, reference_type, reference_id) prevents any double-crediting.

3.5 Nav Balance Widget

A RewardBalanceWidget component was added to AppLayout — visible on every authenticated page for users with the module.redemption_store permission.

  • Shows available balance only (not reserved)
  • For multi-currency holders: primary currency shown, tooltip expands all

  • Clicking the widget navigates to /redemption-store

4. F-02 — Redemption Catalog Done

What this delivered: A three-tier catalog system. Platform Admin defines what exists globally. Client Admin decides what their partners can see, at what minimums, in which modes, and in which regions. Partners browse a filtered, currency-aware catalog with shortfall indicators.

4.2 Platform Admin — Global Catalog

RedemptionCatalogItem is a platform-level entity with no client_id — it is not tenant-scoped. This is analogous to how feature flags work. TENX_ADMIN creates items globally; any Client Admin can then enable them for their tenant.

CASH items

Routed to XTRM. Always amount-based. Cannot be returnable. Vendor name never shown.

NON_CASH items

Routed to Xoxoday. Gift cards, vouchers, etc. May be returnable. Provider item ID required before activation.

A background sync job (XoxodaySyncJobService) periodically pulls Xoxoday's catalog. Items that Xoxoday removes are automatically deactivated (isActive = false), immediately hiding them from all tenants.

4.3 Client Admin — Tenant Configuration

Client Admin never touches the global item. They create a ClientCatalogItemConfig row (one per item per tenant) to control what their partners see.

4.4 Regional Configuration

For each catalog item, Client Admin can further restrict availability by region using ClientCatalogRegionConfig. The fallback logic is a three-tier check:

flowchart LR Q{"Does ClientCatalogRegionConfig\nexist for this\n(tenant, item, region)?"} Q -->|Yes| A["Use its enabled value"] Q -->|No| B{"Does ClientCatalogItemConfig\nexist for this\n(tenant, item)?"} B -->|Yes| C["Use ClientCatalogItemConfig.enabled\n(applies to all regions)"] B -->|No| D["Item NOT visible"] style A fill:#dcfce7,stroke:#22c55e style D fill:#fee2e2,stroke:#ef4444

Hard constraint: A Client Admin cannot enable an item for a region that is NOT in the item's geographicScope (set by Platform Admin). Attempting this returns HTTP 422.

4.5 Processing Modes

INSTANT

Sent to vendor immediately after submission (or after approval if APPROVAL_REQUIRED). DEBIT recorded on webhook confirmation.

BATCH

Queued with a scheduledBatchDate. Processed by batch job on tenant's configured cadence (daily or weekly). Partner sees next batch date.

APPROVAL_REQUIRED

Held in PENDING_APPROVAL. Balance stays reserved. Vendor handoff only after a CLIENT_ADMIN approves (F-04).

4.6 Partner Catalog Browse

When a partner opens the Redemption Store, the catalog is assembled through this filter pipeline:

  1. 1Start with all globally active items (isActive = true)
  2. 2Keep only items the Client Admin has enabled for this tenant (ClientCatalogItemConfig.enabled = true)
  3. 3Apply regional filter — three-tier fallback (see above)
  4. 4Keep only items whose currencyId matches a currency the partner holds a wallet for
  5. 5For each item: compute canAfford (available balance ≥ effective min wallet balance) and shortfallAmount
  6. 6Sort: group by currency, then NON_CASH by regional relevance, CASH by min amount ascending

4.7 Shortfall Indicator

Items the partner cannot yet afford are not hidden — they are shown with a ShortfallBadge displaying how much more balance is needed. The "Redeem" button is disabled for these items. This is intentional: it motivates partners to keep earning.

Important: minWalletBalance (the threshold check) is NOT shown to partners. It is deliberately hidden to prevent gaming. Partners only see the shortfall amount, not the raw threshold.

4.8 Effective Value Resolution

When building a catalog item's response for a partner, the platform always resolves effective values using a COALESCE pattern — override takes precedence, global default is the fallback:

FieldEffective Value Formula
Processing modeCOALESCE(config.processingModeOverride, item.defaultProcessingMode)
Min transaction amountCOALESCE(config.minTransactionAmountOverride, item.defaultMinRedemptionAmount)
Min wallet balanceCOALESCE(config.minWalletBalanceOverride, 0)
Return window daysCOALESCE(config.returnWindowDaysOverride, item.defaultReturnWindowDays)

These effective values are also what F-03 (Redemption Flow) uses at submission time for validation.

5. F-03 — Redemption Flow In Progress Vendor Blocked

What this delivers: End-to-end redemption submission — partner selects a catalog item, wallet balance is reserved atomically, request is routed to XTRM (cash) or Xoxoday (non-cash), vendor webhook confirms fulfillment and finalises the ledger. All 3 processing modes supported. Default cap of 10 in-flight redemptions per partner (configurable per tenant).

6. How Features Connect

The three completed or in-progress features form a layered dependency chain. F-01 provides the wallet and ledger engine. F-02 builds the catalog and browse experience on top of wallet balances. F-03 uses both — pulling effective catalog values at submit time and calling WalletService to reserve, debit, or release the balance.

Once F-03's vendor-blocked stories resolve, it will unlock F-04 (approval queue for APPROVAL_REQUIRED redemptions) and F-05 (transaction history querying RedemptionRequest records). Both are independent and can be built in parallel.
